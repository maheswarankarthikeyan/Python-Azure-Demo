üéØ Workshop Methodology: Best Practices, Practical, Documentation-Backed

This workshop follows industry best practices and Microsoft's recommended patterns for enterprise AI implementations across healthcare, banking, and financial services. The recommendations, architecture patterns, and Azure service selections are based on proven approaches documented in Microsoft's Cloud Adoption Framework and validated through real-world deployments.

The fundamental principle guiding this methodology: "Start with the business problem, not the technology." Research shows that AI projects often fail when teams jump to solutions before understanding the problem. This structured approach ensures the right solution is built for the right problem.

The Three Pillars of Our Methodology

1
Industry Best Practices

Principle: Architecture patterns recommended in this workshop are based on Microsoft's documented best practices and industry-validated approaches for regulated industries.

Example: RAG (Retrieval-Augmented Generation) is recommended for clinical note summarization because fine-tuned models present challenges in healthcare environments:

  • Inability to cite sources (audit requirement for HIPAA)
  • Risk of hallucinations about patient data (unacceptable in clinical settings)
  • Expensive retraining when medical terminology evolves

RAG addresses these challenges: it grounds responses in actual patient notes, provides citations for audit trails, and requires no model retraining.

Principle: This workshop provides specific configurations, code samples, and Azure service settings that can be directly applied to real implementations.

Azure Service Selection with Reasoning

Azure OpenAI ‚Äî GPT-4 Turbo

Why GPT-4 Turbo specifically?

  • 128K context window: Clinical histories can be long. GPT-4 Turbo handles 128K tokens vs. 8K for GPT-3.5. This means we can include more relevant notes in context.
  • 3x faster than GPT-4: Our <5 second latency requirement is tight. GPT-4 Turbo with streaming typically responds in 2-4 seconds.
  • Better instruction following: Clinical summarization requires precise formatting (sections, citations). GPT-4 Turbo follows complex instructions more reliably.
  • HIPAA eligible: Azure OpenAI is covered under Microsoft's BAA, unlike OpenAI's direct API.

Configuration:

  • Model: gpt-4-turbo (1106-preview)
  • Region: East US (HIPAA BAA, lowest latency for East Coast)
  • TPM: 80,000 tokens/minute (sized for 50 concurrent users)
  • Temperature: 0.3 (low for factual accuracy, not 0 to allow readable prose)
  • Streaming: Enabled (perceived latency improvement)

Azure AI Search ‚Äî Hybrid Search

Why hybrid search (vector + keyword)?

  • Vector search alone misses exact matches: A search for "ICD-10: I20.9" needs exact keyword matching. Vector search might return semantically similar but wrong codes.
  • Keyword search alone misses semantic meaning: A search for "chest pain" should find notes mentioning "angina" or "cardiac discomfort." Keyword search misses these.
  • Hybrid combines both: We get semantic understanding AND exact matching. Best of both worlds.

Configuration:

  • SKU: Standard S2 (handles 2.3M documents with room to grow)
  • Replicas: 2 (99.9% SLA, high availability)
  • Partitions: 2 (storage for 2.3M notes + embeddings)
  • Semantic ranking: Enabled (improves relevance of top results)

Compliance Implementation

HIPAA Compliance Checklist

Every element of this architecture is designed for HIPAA compliance:

  • Business Associate Agreement (BAA): Signed with Microsoft covering Azure OpenAI, AI Search, Blob Storage, Monitor. Available in Azure Trust Center.
  • Data Residency: All resources deployed in US regions only (East US). Azure Policy enforces this.
  • Encryption at Rest: Customer Managed Keys (CMK) via Azure Key Vault. Keys rotated every 90 days.
  • Encryption in Transit: TLS 1.2+ enforced. TLS 1.0/1.1 disabled in storage account settings.
  • Network Isolation: Private endpoints for all services. No public internet access to PHI. ExpressRoute connection to hospital network.
  • Audit Logging: All AI queries logged to Log Analytics. 7-year retention (HIPAA minimum). Immutable storage for tamper-proof audit trail.
  • Access Control: Azure Entra ID with RBAC. Only clinicians with "ClinicalAI.Reader" role can access summaries. MFA required.
  • Minimum Necessary: Summaries include only information relevant to current encounter. Full notes available via citation links for verification.

Why We Don't Do AI Risk Scoring (Yet)

You might wonder: "Why not have AI score the loan risk too?" Here's our reasoning:

  • Fair lending laws are strict: ECOA and HMDA require that credit decisions not discriminate based on protected classes (race, gender, age, etc.). AI models can inadvertently encode bias.
  • Explainability requirements: If you deny a loan, you must explain why. "The AI said no" is not acceptable. You need specific factors.
  • Regulatory scrutiny: Bank examiners (OCC, FDIC) are increasingly focused on AI in credit decisions. A poorly implemented AI scorer can result in consent decrees.
  • Recommended approach: Start with extraction (low risk, high value). Validate the technology works. Then, in Phase 2, consider AI-assisted scoring with proper bias testing, explainability, and legal review.
