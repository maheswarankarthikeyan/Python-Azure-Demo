Banking Use Case Examples

⚡ Azure Functions - Best For:

  • Fraud Detection API: HTTP trigger, <200ms response
  • Document Processing: Blob trigger, extract data
  • Transaction Validation: Queue trigger, validate rules
  • Daily Reports: Timer trigger, generate reports
  • Notification Service: Event Grid trigger, send alerts

🐳 Container Apps - Best For:

  • Credit Scoring Model: ML inference, custom dependencies
  • Loan Processing Service: Complex workflow, >10 min
  • Customer Portal API: Full web application
  • Data Pipeline Worker: Long-running batch jobs
  • Legacy App Modernization: Containerize existing apps
💡
Pro Tip: Start with Functions for simplicity. Move to Container Apps when you need custom runtimes, longer execution, or complex dependencies.

🏗️ Architecture: Azure Functions Event-Driven System

SecureBank Fraud Detection Architecture

🏦 SecureBank Implementation

Real-time Fraud Detection Flow:

  1. Payment gateway sends transaction → HTTP trigger → Fraud Detection Function
  2. Function calls ML model endpoint for risk score
  3. Queries Cosmos DB for customer history and patterns
  4. If fraud detected, publishes to Service Bus → Notification Service
  5. All telemetry flows to Application Insights for monitoring
  6. Alerts trigger if error rate > 1% or latency > 500ms

Banking Use Cases

📨 Service Bus

  • Loan Processing: Sequential steps, guaranteed delivery
  • Payment Orders: FIFO processing required
  • Account Updates: Reliable state changes

⚡ Event Grid

  • Account Created: Notify multiple systems
  • Transaction Completed: Real-time notifications
  • Fraud Detected: Broadcast to all subscribers

🔄 Logic Apps

  • Loan Approval: Multi-step workflow with approvals
  • Customer Onboarding: Orchestrate multiple services
  • Compliance Reporting: Scheduled data aggregation
💡
Pro Tip: Use Service Bus for commands (do something), Event Grid for events (something happened), Logic Apps for workflows (orchestrate multiple steps).

🏗️ Architecture: Container Apps Deployment Pipeline

SecureBank Credit Scoring Service - CI/CD to Production

🏦 SecureBank Credit Scoring Deployment

Production Deployment Strategy:

  • Build: Docker image with XGBoost model + FastAPI (450MB)
  • Registry: Push to Azure Container Registry (securebank.azurecr.io)
  • Deploy: Container Apps with canary deployment (10% traffic initially)
  • Monitor: Application Insights tracks latency, errors, business metrics
  • Scale: Auto-scale 2-20 replicas based on CPU, HTTP requests, queue length
  • Rollout: Gradual increase over 7 days (5% → 25% → 50% → 100%)
  • Rollback: Instant rollback to v1.0 if issues detected

Deployment Strategy Comparison

StrategyDowntimeRollbackCostRiskBest For
Rolling UpdateBriefSlowLowMediumNon-critical updates

Banking Deployment Scenarios

🔵🟢 Blue-Green - Use When:

  • Core Banking System: Zero downtime required
  • Payment Processing: Instant rollback critical
  • Regulatory Changes: Must deploy at specific time
  • Database Migrations: Need to test before cutover

Example: Deploy new payment engine - test on green, switch traffic, rollback if issues

🐤 Canary - Use When:

  • ML Model Updates: Validate accuracy gradually
  • API Changes: Monitor error rates incrementally
  • UI Updates: Test with small user percentage
  • Performance Tuning: Validate improvements safely

Example: Deploy new fraud model - 5% → 25% → 50% → 100% over 7 days

🅰️🅱️ A/B Testing - Use When:

  • Feature Experiments: Test new features with segments
  • UX Changes: Compare conversion rates
  • Algorithm Variants: Test different approaches
  • Personalization: Route by customer segment

Example: Test two loan approval algorithms - route by customer type

🔄 Rolling Update - Use When:

  • Minor Updates: Bug fixes, patches
  • Non-critical Services: Internal tools
  • Cost Sensitive: Can't afford 2x infrastructure
  • Low Risk: Well-tested changes

Example: Update internal reporting service - low risk, cost-effective

⚠️
Banking Best Practice: Always use Blue-Green or Canary for customer-facing services. Never use rolling updates for payment processing or core banking systems.
💡
Pro Tip: Start with Canary for most deployments. It provides safety with minimal cost. Use Blue-Green only when instant rollback is absolutely critical.

🐳 Dockerizing LLM Applications for Azure Container Apps

Complete Architecture & Best Practices

Multi-Stage Docker Build Strategy

📦 Layer Optimization

# Stage 1: Base Image
FROM python:3.11-slim AS base
WORKDIR /app
RUN apt-get update && apt-get install -y \
    gcc g++ && rm -rf /var/lib/apt/lists/*

# Stage 2: Dependencies
FROM base AS dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Stage 3: Model Layer (cached separately)
FROM dependencies AS model
COPY models/ /app/models/
# XGBoost model: 250MB

# Stage 4: Application
FROM model AS application
COPY app/ /app/
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]

Container Registry Workflow

✅ Best Practices

  • Use multi-stage builds
  • Cache model layer separately
  • Minimize image size
  • Security scan with Trivy
  • Use .dockerignore

🔐 Security

  • Azure Key Vault for secrets
  • Managed Identity auth
  • Private endpoints
  • Image vulnerability scanning
  • Non-root user execution

📊 Optimization Results

  • Before: 450MB
  • After: 180MB (60% reduction)
  • Build time: 3min → 45sec
  • Deploy time: 2min → 30sec
  • Cost savings: 40%
