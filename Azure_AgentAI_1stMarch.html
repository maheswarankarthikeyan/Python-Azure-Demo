🏢 The World That Brought Us Here

Real Customer Brief — Fortune 500 Insurance Carrier, Midwest US, 2024
From 40 manual workflows to one agentic platform — in 6 months

A large insurance carrier came to our Microsoft team with a problem that looked simple on the surface: they had 40+ disconnected workflows — underwriting, claims triage, policy renewals, compliance checks, reinsurance lookups — each run by separate teams, separate RPA bots, and separate integration contracts. The cost of coordination alone exceeded $12M annually. Errors in handoffs cost another $4M in rework. The core failure: no system could reason across workflow boundaries.

They didn't need better automation. They needed agents that could reason, delegate, recover from failure, and explain their decisions to a regulator. This session is built entirely around what we built for them — and what we wish we'd known at the start.

Every pattern, every architecture decision, every tradeoff we discuss today maps to something we actually had to solve, argue about in a whiteboard session, and defend in a production incident review.

The most important thing I'll tell you today: What separates an "AI workflow" from an "Agentic System" is not the technology — it's the reasoning loop. A workflow is deterministic: input → predefined steps → output. An agentic system is goal-directed: the agent decides which tools to call, in what order, whether to retry, when to escalate, and how to explain its choices. This single distinction changes everything about how you architect, test, monitor, and govern your systems.

The Anatomy of an Agent — The Building Block You Must Own

An agent in Azure is not just a prompt. It is a reasoning loop — a continuous cycle of observation, thinking, acting, and reflecting. The technical name for this is the ReAct loop (Reason + Act), but the practical shape is always the same four components:

🎯 Goal

Not an instruction — a desired outcome state. "Onboard employee ID 4421 fully by end of day" is a goal. "Call the HR API" is an instruction. Goals allow the agent to reason about alternative paths when the first approach fails.

🔧 Tools (Function Calling)

APIs, databases, code interpreters, search engines, other agents. Every tool is registered with a schema that describes what it does, its parameters, and expected outputs. GPT-4o and the Azure AI Foundry Agent Service handle tool calling natively. The agent decides which tool, when, with what arguments.

🧩 Memory

Short-term: The conversation context window — what happened this session. Limited by token count (128k for GPT-4o, more for some models).
Long-term: Azure AI Search vector store, Cosmos DB, Redis — persistent facts and episodic history that survive context resets.
Working memory: Structured state in Durable Functions or Azure Table Storage — the agent's "scratchpad" for a multi-step task.

🔀 Orchestration Logic

The rules for when to call a tool, when to stop, when to ask for clarification, when to spawn a sub-agent, and when to escalate. This is where most agent failures happen — and where architects earn their salary.

Single Agent vs Multi-Agent — When Do You Need a Crew?

✅ Stay with a Single Agent When:

  • The task fits in one LLM context window
  • One domain of expertise is sufficient
  • You need low latency (<3s end-to-end)
  • The task is well-defined with predictable steps
  • You're early in your AI maturity journey
  • Cost control is paramount (single agent = ~5-10x cheaper per query)

Example: A customer FAQ bot. One agent with RAG against a knowledge base.

🤝 Go Multi-Agent When:

  • Task complexity exceeds one agent's reasoning budget
  • Different steps require different specialist expertise (legal + financial + technical)
  • Steps can be parallelized for speed (fan-out pattern)
  • You need independent quality verification (critique agent)
  • The workflow has human approval gates between phases
  • Different agents need different tool permissions (principle of least privilege)

Example: Insurance underwriting. HR agent, legal agent, risk calculation agent, compliance agent.

The Most Common Architect Mistake: Reaching for multi-agent complexity when a well-prompted single agent with good tools would work fine. Multi-agent adds latency, cost, failure points, and debugging complexity. Always prototype single-agent first. Escalate to multi-agent only when you hit a concrete ceiling.

🔁 Pattern 1 — Sequential (Pipeline) Pattern

➡️Sequential PatternApplication: New Employee Onboarding Crew

What Is It?

The Sequential pattern is a linear pipeline of specialized agents where each agent completes its task, produces an output, and hands off to the next agent in the chain. The output of agent N is the input of agent N+1. Think of it as an assembly line where each station has a specialist — and the product is only complete when it passes through every station in order.

This is the simplest multi-agent pattern and the right starting point for most enterprise process automation. The predictability and debuggability of a linear pipeline is a feature, not a limitation — especially when you're presenting to a regulator.

🏢 Case Study — New Employee Onboarding Crew
Contoso Insurance: Automating Day-One from 3 days to 47 minutes

Business Problem: When a new hire's record appeared in Workday, a 3-day manual process kicked off: HR validates eligibility, IT provisions accounts, Security assigns badges, Compliance runs background verification, and a manager receives a welcome package. Each step was done by a different team, in a different system, with emails as the handoff mechanism. New hires regularly had no laptop access for their first 2 days.

Solution: A 5-agent sequential crew, each agent owning exactly one domain. Total wall-clock time reduced from 3 days → 47 minutes for standard hires.

🧑‍💼 Pattern 2 — Supervisor (Hub-and-Spoke) Pattern

👔Supervisor PatternApplication: Sales Intelligence Crew — Sales Analyst + Market Researcher → Supervisor

What Is It?

The Supervisor pattern introduces a coordinator agent (the Supervisor) that does not perform domain work itself — it delegates, monitors, and synthesizes. The supervisor receives a complex goal, decomposes it into sub-tasks, dispatches specialist agents (workers) to execute in parallel or sequence, collects their outputs, resolves conflicts, and produces a unified answer.

This is the pattern that scales the most naturally. Adding a new specialist to the crew is as simple as registering a new worker with the supervisor and teaching the supervisor when to use it. No pipeline refactoring needed.

The key architectural insight: the supervisor is also an LLM agent. It uses reasoning to decide delegation strategy — not hardcoded routing logic. This is what makes it adaptive to novel inputs.

💼 Case Study — Sales Intelligence Crew
Contoso's Sales VP wants a daily intelligence brief on a target account — in under 5 minutes

Business Problem: A sales team targeting a Fortune 500 prospect needed, before every executive meeting: (1) the prospect's latest financial performance, (2) competitive positioning in their market, (3) Contoso's prior engagement history, (4) news and risk signals from the past 7 days, and (5) a recommended talking-point strategy. Previously this took a junior analyst 4 hours. Quality was inconsistent. Meetings were often postponed because the brief wasn't ready.

Solution: A Supervisor agent that coordinates a Sales Analyst agent and a Market Researcher agent, running in parallel, then synthesizes their outputs into a structured brief.

Probe #3 — Supervisor Failure Modes: The Market Researcher agent returns a result, but it was hallucinated — it made up a news story about the target company's bankruptcy that never happened. The Supervisor incorporates it into the brief. The Sales VP presents it to the client. Three questions:

When Supervisor Shines vs. When It Struggles

✅ Supervisor Pattern Wins When:

  • Sub-tasks can run in parallel (2-4x speed vs sequential)
  • Results from different domains need conflict resolution
  • You need to add/remove specialists without restructuring the pipeline
  • Worker agents need different tool permissions (security boundary per specialist)
  • The task is open-ended enough that the supervisor needs to dynamically choose which workers to invoke

⚠️ Supervisor Pattern Struggles When:

  • Steps are strictly sequential with hard data dependencies (use Sequential instead)
  • The supervisor itself becomes a single point of failure — no fallback
  • Synthesis quality depends on the supervisor's reasoning — junk supervisor prompt = junk output regardless of worker quality
  • Cost is a hard constraint: 2 workers + supervisor = 3x LLM calls minimum
  • Workers return ambiguous or contradictory results that require human judgment to reconcile

🔄 Pattern 3 — Reflection (Group Chat / Round Robin) Pattern

✍️Reflection PatternApplication: Advertising Copy Creation — Writer Agent + Critic Agent

What Is It?

The Reflection pattern introduces an iterative quality improvement loop where one agent produces output and a second agent critiques it against a rubric — then the first agent revises, and the loop continues until a quality threshold is met or max rounds is reached. In AutoGen and Semantic Kernel this is implemented as a Round Robin Group Chat.

This pattern is designed for tasks where quality matters more than speed, where the output is subjective or complex enough to benefit from multiple perspectives, and where you can define a clear rubric for what "good" looks like. Think: legal document review, medical report generation, code review, creative content.

The critical design decision: The stopping condition. Too few rounds = low quality. Too many = cost explosion + diminishing returns. In practice, 2-3 rounds typically capture 80% of the quality improvement. Round 4+ is usually minor polish that isn't worth the token cost.

🎨 Case Study — Advertising Copy Creation Workflow
Contoso's Marketing Team: Generating Compliant, Brand-Perfect Insurance Ad Copy at Scale

Business Problem: A marketing team needed to generate compliant insurance advertisement copy for 12 regional markets, each with different regulatory requirements and brand tone guidelines. A human copywriter + compliance reviewer pair took 3 hours per piece. They needed to generate 240 pieces per month. That's 720 human-hours they didn't have.

Solution: A Writer Agent + Critic Agent reflection loop. Writer generates copy per brand guidelines. Critic evaluates against a compliance rubric and brand scorecard. Writer revises. Maximum 3 rounds. If round 3 still fails compliance, escalate to a human reviewer (not abandon — escalate).

⚙️ Workflow Orchestration — The Infrastructure That Makes Agents Production-Worthy

🔧Orchestration EnginesDurable Functions · Logic Apps · State Machines · Event-Driven · Retries · Branching
The production reality check: Everything we've covered so far about agents assumes the LLM calls succeed on the first try. They don't. GPT-4o returns 429 rate limit errors at 2am when your batch runs. An external API returns 504 after 30 seconds. A Durable Function host restarts mid-workflow. An agent loops 47 times before hitting its max_iterations cap. Without a durable orchestration layer underneath your agents, you are one infrastructure hiccup away from a corrupted, inconsistent, un-restartable workflow. This is not theoretical — I've seen it happen in production at 3am.
Azure Durable Functions
Stateful serverless · Long-running agent workflows · Checkpoint & replay · Human-in-the-loop

Durable Functions provides durable execution — workflows that survive process restarts, network failures, host recycling, and arbitrary delays. The secret: every step's inputs and outputs are persisted to Azure Storage as events. If the host crashes mid-workflow, it replays from the last checkpoint. Your agent workflow code looks sequential; the runtime makes it fault-tolerant.

For agentic AI, this means: agent state is persisted automatically, retries are built-in at the infrastructure level, and you can pause a workflow waiting for a human approval or external event for hours, days, or weeks without consuming compute resources during the wait.

The 5 Durable Patterns for Agents

1
Function Chaining
Sequential agent pipeline with automatic checkpointing after each step. Perfect for the Sequential onboarding pattern.
2
Fan-Out / Fan-In
Supervisor spawning N parallel worker agents simultaneously (Task.WhenAll), collecting results when all complete or timeout fires.
3
Human-in-the-Loop
WaitForExternalEvent suspends the orchestration until a human approves via a webhook. The workflow is frozen, zero compute cost during the wait. Perfect for compliance gates.
4
Eternal Orchestration
A monitoring agent that wakes up on a schedule or external event, runs, then calls ContinueAsNew to restart itself. The agentic equivalent of a daemon process.
5
Saga / Compensation
For sequential patterns: if step N fails, execute compensating actions for steps 1 through N-1 in reverse order. The distributed transaction pattern without a transaction coordinator.

Retry Policy — The Detail That Saves Your 3am

// Durable Functions retry — exponential backoff, catches 429 + 503
var retryPolicy = new RetryOptions(firstRetryInterval: FromSeconds(5), maxAttempts: 3) {
    BackoffCoefficient = 2.0, // 5s → 10s → 20s
    Handle = ex => ex is HttpRequestException || ex.Message.Contains("429")
};
await ctx.CallActivityWithRetryAsync("SalesAnalystAgent", retryPolicy, input);
Never retry on 400 Bad Request or 401 Unauthorized. Retrying on 429 (rate limit) and 503 (transient) is correct. Retrying on auth errors will hit your max_attempts instantly and waste time. Handle error categories, not generic exceptions.

If the activity doesn't complete by deadline → transition to STEP_FAILED, not infinite wait. I've seen hung Durable instances sitting for 11 days because someone forgot a timeout on one state.

It is emphatically not an agent orchestrator — it is the event bus and integration hub that feeds agents with triggers and routes their outputs to downstream systems. With 1,000+ connectors (SAP, Salesforce, Dynamics 365, ServiceNow, SharePoint, Outlook, Teams, Workday), Logic Apps is what makes agents context-aware without you writing custom integrations for every enterprise system.

🔍 Implementing Adaptive RAG with Multi-Agent Systems

📚Adaptive RAGDynamic retrieval routing · Query intelligence · Multi-source synthesis · Agent-driven search

From Standard RAG to Adaptive RAG — What Changes?

Standard RAG: query → embed → top-K vector search → stuff chunks into context → generate. Simple, predictable, often wrong. The failure mode: you retrieve the top 5 most semantically similar chunks regardless of whether they actually answer the question. Complex multi-hop questions (involving regulations, time ranges, relationships between entities) fail because no single chunk contains the full answer.

Adaptive RAG adds a query intelligence and quality control loop around retrieval. The agent first decides what type of question it is, selects the retrieval strategy accordingly, retrieves, evaluates the relevance of what it got, and can requery with a reformulated question if the first retrieval was insufficient. It's retrieval as an agentic loop, not a one-shot operation.

🏦 Case Study — Contoso's Policy Intelligence Agent
Answering Complex Insurance Policy Questions That Span Multiple Knowledge Domains

Problem: Insurance underwriters needed to answer questions like: "Does our current reinsurance treaty for Midwest flood events cover losses caused by levee failures due to deferred maintenance, given the new FEMA 2024 flood zone remapping?" This requires: internal policy documents (specific treaty language), FEMA regulatory guidance (external, updated), historical loss case precedents (internal claims database), and real-time flood zone boundary data (live API). Standard RAG with a single vector index failed because these four knowledge sources have completely different characteristics: update frequency, format, authority level, and retrieval optimal strategy.

Solution: Adaptive RAG with a query routing agent that classifies question type, selects retrieval strategy per source, evaluates chunk relevance before passing to the generation model, and loops if relevance falls below threshold.

📊 Evaluating Agents — The Discipline That Makes AI Trustworthy

🔬Agent EvaluationTool call accuracy · Goal accuracy · Golden datasets · AI Foundry Evaluation SDK

Why Agent Evaluation Is Different from Model Evaluation

Evaluating a static model is straightforward: benchmark dataset, measure accuracy, done. Evaluating an agent is fundamentally different because agents are goal-directed, tool-using, multi-step systems. The model itself might be excellent, but if the agent calls the wrong tool, or calls the right tool with wrong parameters, or calls the right tool correctly but doesn't know when to stop — the system fails. You need to measure all three layers independently.

Layer 1
Tool Call Accuracy
Does the agent call the RIGHT tool with the RIGHT parameters at the RIGHT time?
Layer 2
Goal Accuracy
Does the agent achieve the stated goal end-to-end, including when tools fail or return unexpected results?
Layer 3
Safety Compliance
Does the agent stay within guardrails? Does it escalate appropriately? Does it refuse harmful requests?

Tool Call Accuracy — Measuring the Right Skill

Tool call accuracy asks: given a user request, does the agent invoke the correct function (or sequence of functions) with the correct arguments? This is measurable and automatable. You need a golden dataset: a set of test inputs with expected tool call sequences and expected argument values.

🛡️ Safe Interaction Design and Fallback Routing

🔐Safe Interaction DesignSafety layers · Prompt shields · Guardrails · Escalation tiers · Fallback routing
My rule for every enterprise agentic system: Safety is not a feature you add before launch. It's a design constraint you enforce from the first architecture whiteboard. An agent that can take actions (send emails, write to databases, call APIs, execute code) is a system that can cause harm at scale if safety is wrong. The blast radius of a misconfigured agentic system is orders of magnitude higher than a misconfigured chatbot.

The Five Safety Layers — Defense in Depth for Agents

Agent Safety Architecture — Defense in Depth
LAYER 1 — INPUT SHIELD (Before the Agent Sees the Request)

Azure AI Content Safety — Prompt Shield: Detects prompt injection attempts, jailbreak patterns, and policy-violating inputs before they reach the agent. For enterprise agents with tool access: this layer must stop malicious inputs that could cause the agent to exfiltrate data or take unauthorized actions. ShieldPrompt API call adds ~50ms — worth every millisecond.

LAYER 2 — TOOL PERMISSION LAYER (What the Agent is Allowed to Do)

Principle of Least Privilege per Agent: Each agent should have the minimum tool permissions needed for its role. The HR Validation Agent should be able to READ Workday — not write, not delete. The Identity Agent needs Graph API write for provisioning but not admin role management. Implement RBAC at the tool/API level, not just at the agent level. If an agent is compromised or misbehaves, the damage is bounded by its tool permissions.

LAYER 3 — AGENT BEHAVIOR GUARDRAILS (System Prompt Constraints)

System prompt constraints enforced at the model level: Every agent must have explicit rules in its system prompt: "Never send emails to external domains without explicit user confirmation." "Never delete records — always soft-delete and log." "If a user requests PII of another employee, refuse and escalate." These constraints must be tested adversarially — assume attackers will try to override them via crafted inputs.

LAYER 4 — HUMAN-IN-THE-LOOP GATES (High-Stakes Decision Escalation)

Mandatory human approval for high-stakes actions: Define a "high-stakes action" list for your domain. In insurance: any coverage decision over $500K, any denial of a claim, any policy interpretation that sets precedent. These must pause the workflow and route to a human via Teams approval or ServiceNow ticket. Use Durable Functions WaitForExternalEvent. The agent recommends; the human decides. This is your regulator-satisfying safety valve.

LAYER 5 — OUTPUT SAFETY FILTER (Before the Response Reaches the User)

Azure AI Content Safety — Output screening: Screen every agent output for: PII leakage (social security numbers, account numbers surfaced inappropriately), harmful content, and prompt injection in agent-generated content (an agent that generates content consumed by another system is a vector for cascading injection). Log every flagged output with full trace for audit.

Fallback Routing — Designing the Escalation Tiers

A fallback routing strategy answers: when the agent can't or shouldn't complete a task, where does it go? This is not optional. Every agent will encounter situations it cannot handle correctly. The question is whether you've designed a graceful, auditable path — or whether the agent will silently fail, hallucinate, or take a dangerous action.

Escalation Tiers — Fallback Routing Design
TIER 0
Agent
Agent handles it autonomously. High confidence, within tool permissions, task type well-covered in golden dataset. Goal accuracy ≥ 88%. This is the target state for ≥85% of requests.
↓ if low confidence or forbidden action requested
TIER 1
Clarify
Agent asks the user for clarification. Ambiguous input, missing required context. Agent presents 2-3 clarifying questions. Continues after user responds. Appropriate for ≥8% of requests.
↓ if task type unknown, policy boundary, or confidence below threshold
TIER 2
Specialist
Route to a specialist human. Domain expert who can resolve the case and optionally provide labeled feedback to improve the golden dataset. Target ≤5% of requests. If this tier is processing >10%, your agent needs more training data or better routing logic.
↓ if safety violation detected, legal boundary, or executive-level decision
TIER 3
Hard Stop
Agent refuses and logs with full trace. Prompt injection detected, forbidden action, regulatory boundary crossed. No partial action taken. Alert sent to security team. Full context preserved for audit. This must never fail silently. Target <1% but 100% reliability required.
Track your escalation tier distribution weekly. If Tier 2 (specialist) starts rising, it's a signal: either your agent's training data doesn't cover new request types, or a new regulatory requirement has created a blind spot. Escalation rate is a leading indicator of agent health, not just a support metric.

👁️ Observability — Seeing Inside Your Agent Workflows

📡ObservabilityAzure Monitor · Application Insights · OpenTelemetry · Agent-specific metrics · Alerting

Why Agent Observability Is Different from API Observability

Traditional API observability answers: did the endpoint respond? How long did it take? What was the error rate? Agent observability must answer a much harder set of questions:

  • Why did the agent call tool X instead of tool Y? (Reasoning transparency)
  • Which step in a 5-agent pipeline caused the output quality to degrade? (Attribution across agents)
  • Is the agent's goal accuracy trending down after a system prompt change? (Regression detection)
  • Which user queries are hitting the escalation tier most often? (Coverage gap detection)
  • What is the token cost per agent run, by workflow type? (FinOps for AI)

Answering these questions requires structured tracing across the entire agent reasoning loop — not just HTTP request/response logs.

The Azure Observability Stack for Agents

Agent Observability Architecture — Azure Monitor + Application Insights + OpenTelemetry

What You Instrument

Every LLM call: input tokens, output tokens, model, temperature, latency. Trace ID links all calls within one agent run. Custom dimension: "agent_name", "step_number", "tool_selected".

Every tool invocation: tool name, arguments (sanitized — remove PII before logging), execution time, success/failure, result size. Failures logged with error code + retry count.
