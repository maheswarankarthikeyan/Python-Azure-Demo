What is an AI Agent? (Architecture-First Definition)
Azure AI Agent Service · Autonomous Loop · Tool Use · Memory
Core Concept Hour 1

An AI agent is not a chatbot. A chatbot answers a question. An agent completes a goal by deciding what actions to take, using tools to gather information and execute actions, and adapting based on feedback — without a human directing every step.

The Architect's Definition of an Agent (memorise this):
Agent = LLM + Memory + Tools + Autonomy Loop

LLM = the reasoning engine (GPT-4o via Azure OpenAI)
Memory = context of what has happened (thread state in Azure AI Agent Service)
Tools = functions/APIs the agent can call (code interpreter, function calling, file search)
Autonomy Loop = the agent decides the sequence — you don't hardcode it

ReAct Pattern — Reason → Act → Observe → Repeat

Semantic Kernel — Enterprise Agent SDK
Microsoft's Production-Grade Orchestration Framework for Agents
Core SDK Production Ready

Semantic Kernel (SK) is what sits between your application code and Azure OpenAI. Think of it as the enterprise middleware for AI — it manages plugins (tools), memory, planner (routing), and chat history. For Contoso, SK is how the Orchestrator Agent decides which sub-agent to invoke.

SK Core Concepts for the Architect

Kernel
The Container
Wires together AI services, plugins, memory. One Kernel per agent process.
Plugin
Tool Definition
Any C# class / Python function annotated with @kernel_function becomes a callable tool.
Planner
Routing Brain
Stepwise Planner creates a plan from a goal. Function Calling Planner uses native tool use.
Contoso Plugin Example — CRM Tool as SK Plugin: 
@kernel_function(description="Get customer mortgage application data from Contoso CRM")
async def get_customer_application(self, customer_id: str) -> str:
    """Returns customer income, credit score, existing liabilities from CRM"""
    response = await self.crm_client.get_async(f"/api/v2/customers/{customer_id}/mortgage")
    return json.dumps(response.json())  # Agent receives this as tool result

Azure API Management as the Agent's Tool Gateway

Every tool the agent calls should go through Azure APIM. This is non-negotiable in an enterprise architecture. APIM gives you: rate limiting (so a runaway agent loop doesn't DDoS your CRM), request/response logging (FCA audit trail), authentication normalisation (the agent uses managed identity; APIM translates to whatever auth the backend needs), and a single point to update tool endpoints without touching agent code.

Architecture Decision Record — APIM Rate Limiting for Agents:
Set per-agent, per-minute rate limits in APIM to cap runaway loops. For Contoso: 60 CRM calls/minute/agent. If an agent loop misfires and tries 300 calls in 10 seconds, APIM returns 429 — the agent fails gracefully instead of crashing the CRM. This is circuit breaking for agentic AI.

Dockerising the LLM Agent for Azure Container Apps
From FastAPI + Semantic Kernel to a production-hardened container
Container Apps Docker

The agent is just a Python (or C#) application with Azure SDK dependencies. Containerising it gives you portability, reproducibility, and the ability to deploy anywhere. The hard parts are: secrets management (never bake API keys into images), warm-up time (LLM SDK initialisation takes 3–8 seconds — this affects cold start SLAs), and health probes (Container Apps needs to know your agent is ready to handle requests).

Contoso Mortgage Agent — Dockerfile (Production Pattern)

# ── Stage 1: Build dependencies (multi-stage for minimal image size) ──
FROM python:3.12-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir --target /deps -r requirements.txt

# ── Stage 2: Runtime (non-root, hardened) ──
FROM python:3.12-slim AS runtime
WORKDIR /app

# Security: non-root user
RUN groupadd -r agentuser && useradd -r -g agentuser agentuser

# Copy dependencies from builder
COPY --from=builder /deps /usr/local/lib/python3.12/site-packages

# Copy application code
COPY --chown=agentuser:agentuser ./src ./src
COPY --chown=agentuser:agentuser ./prompts ./prompts

# Never hardcode secrets — use Managed Identity + Key Vault references
ENV AZURE_CLIENT_ID=""
ENV AZURE_OPENAI_ENDPOINT=""
ENV APPLICATIONINSIGHTS_CONNECTION_STRING=""

# Health probe endpoint (Container Apps liveness check)
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s \
  CMD curl -f http://localhost:8000/healthz || exit 1

USER agentuser
CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "2"]

Azure Container Apps — Key Configuration for LLM Agents

Scaling Rules (KEDA)

Container Apps uses KEDA for event-driven scaling. For an agent API:

  • Scale on HTTP concurrent requests — default for API agents. Set concurrentRequests: 10 per replica.
  • Scale on Service Bus queue depth — for batch agent processing. Each replica picks up one batch job.
  • Min replicas = 1 for Contoso (advisors need sub-second response, not cold starts)
  • Max replicas = 20 with cost alert at 15 (FCA peak — month-end)

Managed Identity — Zero Secrets in Code

The container never sees API keys. Pattern:

  • Container App has a User-Assigned Managed Identity
  • Identity has RBAC: Cognitive Services OpenAI User on Azure OpenAI resource
  • Identity has RBAC: Key Vault Secrets User for CRM API keys
  • Identity has RBAC: Search Index Data Reader on AI Search
  • Code uses DefaultAzureCredential() — works locally (CLI) and in prod (MI) without change

⚠️ The Token Cost Trap — Architects Must Address This Day 1:
GPT-4o charges for every token in the context window, including the conversation history you pass on every request. If your agent thread has 50 rounds of conversation and you pass the full history each time, your costs compound exponentially.

Contoso Mitigation: Implement a sliding window — keep last 10 exchanges + a summarised system memory. Estimated saving: 65% on token costs for long advisory sessions.

PII Logging Strategy for Contoso (FCA Compliant):
NEVER log raw customer PII in Application Insights — it will appear in Microsoft's telemetry infrastructure.
Pattern: Hash identifiers (sha256(customer_id)), tokenise names, store the full prompt+response separately in ADLS Gen2 with encryption-at-rest and private endpoints. App Insights gets the hashed reference. ADLS gets the full payload. Only FCA audit queries access ADLS — not your developers.
Detecting Hallucinations & Model Drift
Groundedness Evaluation · Semantic Drift · Prompt Flow Monitoring · Content Safety
Critical Risk FCA Requirement

In a financial services context, a hallucinated mortgage rate or fabricated FCA rule isn't just a UX issue — it's a regulatory breach. Contoso needs a multi-layer hallucination defence, not just a prompt warning telling the model not to make things up.

The 4-Layer Hallucination Defence Architecture

Layer 1 — Prompt Engineering (Prevention)
Grounding system prompts — Always instruct the model to cite sources from retrieved context. Use a prompt template that says: "If information is not found in the provided context, say 'I don't have that information' — do NOT invent an answer." For Contoso: the Compliance Agent's system prompt includes the retrieved FCA policy excerpts and instructs citation-only responses.
Layer 2 — RAG Groundedness (Grounding)
Azure AI Search retrieval quality gate — Before the LLM generates, check the retrieval score. If the top search result has relevance score < 0.7, don't pass it to the LLM — return "I cannot find a relevant policy for this question." This prevents the LLM from hallucinating when given weak context. Implement as a Semantic Kernel plugin: GroundednessFilterPlugin.
Layer 3 — Online Evaluation (Detection)
Azure AI Foundry Prompt Flow Evaluators running on sampled production traffic. Built-in evaluators: GroundednessEvaluator, RelevanceEvaluator, CoherenceEvaluator. Custom evaluator: MortgageRateAccuracyEvaluator — cross-references any rate mentioned in the agent output against Contoso's live rate database. Scores logged to App Insights as hallucination_risk_score.
Layer 4 — Azure Content Safety (Safety Net)
All agent outputs pass through Content Safety API before delivery to the advisor UI. Content Safety's groundedness detection checks if the output is supported by the provided context. Block or flag responses with groundedness score below threshold. This is your last line of defence — even if layers 1-3 fail, Content Safety can catch blatant hallucinations.

Model Drift Detection Architecture

Drift is different from hallucination. Drift is when the model's behaviour changes over time — either because Azure OpenAI updated the underlying model version, because your data has shifted (new FCA rules not in the index), or because usage patterns have drifted outside the model's training distribution.

Data Drift Signals to Monitor

  • Retrieval coverage rate: % of queries where AI Search returns results with score > 0.7. Declining rate → your index is stale relative to user queries
  • Human escalation rate: If agents defer to humans more often, the model is less confident — a drift signal
  • Topic distribution shift: Use embedding clustering on queries over time. If cluster distribution shifts, usage has moved outside training distribution

Model Version Drift (Azure OpenAI)

  • Azure OpenAI updates model versions automatically unless you pin a deployment to a specific version
  • Always pin production deployments: gpt-4o-2024-11-20 not just gpt-4o
  • Run your evaluation suite against new model versions in a shadow deployment before promoting
  • Log model_version in every App Insights event — this lets you correlate quality changes with version updates retroactively

Top 7 Cost Optimisation Levers for Contoso

Context Window
Implement sliding window (last 10 turns + summarised memory). 40–65% token cost reduction for long sessions.
Model Routing
Use Phi-4 for classification/extraction tasks (routing intent, doc type detection). Reserve GPT-4o for complex reasoning. 70% cost reduction on routed tasks.
Caching
Azure OpenAI Prompt Caching — cache system prompt tokens. For Contoso's 2,000-token system prompt, saves ~30% on every call where system prompt is reused.
PTU Reservation
At >2M tokens/day sustained, Provisioned Throughput Units beat pay-as-you-go by 50–65%. Run for 90 days PAYG to establish baseline before committing.
Scale-to-zero
Container Apps min replicas = 0 outside 8am–8pm. Advisors don't work 24/7. Save ~55% on compute costs for non-production hours.
AI Search SKU
Dev/test on Basic SKU (£60/month). Only provision S2 for production. Use separate search services per environment — don't share prod index with dev.
Log Sampling
App Insights adaptive sampling at 10% for debug telemetry. 100% for errors and custom AI metrics. Reduces App Insights ingestion costs 60–80%.

Identity

✅ Agent configured with max_turns and confidence thresholds. Below threshold → human escalation. No fully autonomous financial decisions — FCA requirement.
